ТОВ «ЕПАМ СИСТЕМЗ»

Вивчення діяльності підприємства

ТОВ "ЕПАМ СИСТЕМЗ" (EPAM Systems) є світовим лідером в галузі розробки програмного забезпечення. Компанія заснована в 1993 році в США і має офіси у більш ніж 35 країнах світу, включаючи Україну.

Напрям діяльності

ТОВ "ЕПАМ СИСТЕМЗ" надає послуги у сфері розробки програмного забезпечення, включаючи розробку програмного забезпечення на замовлення, тестування програмного забезпечення, розробку мобільних додатків та веб-додатків, віртуальну та доповнену реальність, Big Data, Інтернет речей (IoT), аналіз даних та багато іншого.

Шляхи зростання продуктивності праці

Компанія зосереджена на постійному підвищенні продуктивності праці та оптимізації робочих процесів. Для досягнення цих цілей, використовуються такі інструменти, як автоматизація, застосування методологій Agile та DevOps, використання новітніх технологій та розробка власних інструментів для підвищення ефективності роботи.

Склад групи і її кваліфікаційний рівень

ТОВ "ЕПАМ СИСТЕМЗ" має більше 41,000 співробітників у всьому світі, з яких понад 12,500 працюють в Україні. Компанія має дуже високі стандарти відбору співробітників і кваліфікаційний рівень її працівників є дуже високим. Компанія активно співпрацює з університетами та іншими освітніми закладами, щоб підготувати майбутніх спеціалістів і надати їм можливість отримати практичний досвід в галузі розробки програмного забезпечення.

Перелік основних операцій та опис робочого місця

Основні операції компанії включають розробку програмного забезпечення та технічну підтримку для клієнтів у різних галузях. Робочі місця можуть бути різними, в залежності від конкретної ролі, але вони можуть включати написання коду, тестування програмного забезпечення, аналіз та документування вимог клієнта, управління проектами, комунікацію з клієнтами та інші завдання.

Компанія залучається до використання передових технологій, щоб автоматизувати та оптимізувати свою діяльність, включаючи використання штучного інтелекту, машинного навчання, хмарних технологій та інші.

Організація охорони праці на робочому місці

ТОВ "ЕПАМ СИСТЕМЗ" дбає про безпеку своїх співробітників і виконує всі вимоги законодавства щодо охорони праці. Усі співробітники проходять навчання з питань безпеки та охорони здоров'я, а також отримують необхідні інструктажі з роботи на конкретному робочому місці.

Основні засоби по попередженню нещасних випадків

Усі робочі місця обладнані необхідними засобами безпеки, такими як вогнегасники, аптечки першої допомоги, інформаційні таблиці з правилами поведінки у разі виникнення надзвичайних ситуацій. Крім того, підприємство забезпечує своїх співробітників необхідними засобами індивідуального захисту (захисні окуляри, маски тощо).

Таким чином, ТОВ "ЕПАМ СИСТЕМЗ" є сучасним підприємством, яке використовує передові технології та методи для підвищення ефективності своєї роботи. Компанія дбає про своїх співробітників, забезпечуючи їм комфортні умови праці та безпеку.

Завдання

Реалізувати роботу системи Періодичні видання. В системі існує перелік Видань, які згруповані за тематикою. Читач може оформити Передплату на одне або декілька видань. Для переліку видань необхідно реалізувати можливість:

  • - сортування видань за назвою;
  • - сортування видань за ціною;
  • - вибірки видань за певною темою;
  • - пошуку видання за назвою.

Читач реєструється в системі і має особистий кабінет, в якому відображена інформація про видання, на які він підписаний. Незареєстрований користувач не може оформити підписку.

Читач має персональний Рахунок, який він може поповнити. Кошти з рахунку знімаються під час підписки на видання. Адміністратор системи володіє правами:

  • - додавання, видалення і редагування видання;
  • - блокування, розблокування користувача.

Вимоги до реалізації

На основі сутностей предметної області створити класи, які їм відповідають.

Класи і методи повинні мати назви, що відображають їх функціональність, і повинні бути рознесені по пакетам.

Оформлення коду має відповідати Java Code Convention.

Інформацію щодо предметної області зберігати у реляційній базі даних (в якості СУБД рекомендується використовувати MySQL або PostgreSQL).

Архітектура застосунка повинна відповідати шаблону MVC.

Використовуючи сервлети і JSP, реалізувати функціональність, наведену в постановці завдання.

На сторінках JSP застосовувати теги з бібліотеки JSTL та розроблені власні теги (мінімум: один тег custom tag library і один тег tag file).

При розробці використовувати сесії, фільтри, слухачі.

У застосунку повинні бути реалізовані аутентифікація і авторизація, розмежування прав доступу користувачів системи до компонентів програми. Шифрування паролів заохочується.

Застосунок має бути покритим модульними тестами (мінімальний відсоток покриття 40%). Написання інтеграційних тестів заохочуються.

Реалізувати механізм пагінації сторінок з даними.

Застосунок має коректно реагувати на помилки та виключні ситуації різного роду (кінцевий користувач не повинен бачити stack trace на стороні клієнта).

Для доступу до даних використовувати JDBC API із застосуванням готового або ж розробленого самостійно пулу з'єднань.

При реалізації бізнес-логіки необхідно використовувати шаблони проектування: Команда, Стратегія, Фабрика, Будівельник, Сінглтон, Фронт-контролер, Спостерігач, Адаптер та ін.

Використовувати Apache Tomcat у якості контейнера сервлетів.

Реалізувати захист від повторної відправки даних на сервер при оновленні сторінки (реалізувати PRG).

Впровадити у проект журнал подій із використанням бібліотеки log4j.

Код повинен містити коментарі документації (всі класи верхнього рівня, нетривіальні методи і конструктори).

Всі поля введення повинні бути із валідацією даних.

Самостійне розширення постановки задачі по функціональності заохочується! (додавання капчі, формування звітів у різних форматах, тощо).

Використання HTML, CSS, JS фреймворків для інтерфейсу користувача (Bootstrap, Materialize, ін.) заохочується!

Застосунок має підтримувати роботу з кирилицею (бути багатомовним), в тому числі при зберіганні інформації в базі даних:

  • повинна бути можливість перемикання мови інтерфейсу;
  • повинна бути підтримка введення, виведення і зберігання інформації (в базі даних), записаної на різних мовах;
  • 3. в якості мов обрати мінімум дві: одна на основі кирилиці (українська або російська), інша на основі латиниці (англійська).

Реєстрація

Основний успішний сценарій

  • 1. Користувач вводить коректно: ім’я, прізвище, email та пароль. Задані поля мають наступні обмеження:
  • a. імʼя – повинно починатись з однієї великої літери та однієї або більше маленьких латинських літер. Максимальна довжина слова – 50 символів;
  • b. прізвище – повинно починатись з однієї великої літери та однієї або більше маленьких латинських літер. Максимальна довжина слова – 50 символів;
  • c. email – повинен бути валідним відносно синтаксису;
  • d. пароль – повинен містити мінімум: одну цифру, одну латинську літеру у верхньому регістрі, одну латинську літеру у нижньому регістрі, один спеціальний символ. Мінімальна довжина паролю – 8 символів. Максимальна довжина паролю – 32 символа.
  • 2. Система повідомляє про успішно створений обліковий запис.
  • 3. Система перенаправляє користувача на сторінку з авторизацією.

Альтернативний сценарій

  • 1. Користувач ввів невалідні дані або користувач з заданим email вже існує.
  • 2. Система виводить відповідну інформацію про некоректні дані.

Перегляд доступних видань

Основний сценарій

  • Користувач на головній сторінці бачить перед собою n-видань заданих за допомогою пагінації, які містять:
  • • назву
  • • жанр;
  • • опис;
  • • ціну підписки за місяць.
  • Система надає можливість:
  • • сортувати видання за назвою;
  • • сортувати видання за ціною;
  • • вибірки видань за певною темою;
  • • пошуку видання за назвою.

Поповнення персонального рахунку

Основний успішний сценарій

  • 1. Користувач вводить суму, на яку хоче поповнити рахунок.
  • 2. Користувач обирає спосіб оплати: або картою, або за допомогою paypal.
  • 3. Система надсилає запит на транзакцію.
  • 4. Система отримує відповідь про успішну транзакцію.
  • 5. Система поповнює персональний рахунок користувача.

Альтернативний сценарій

  • 1. Система отримує відповідь про невдалу транзакцію.
  • 2. Система сповіщає користувача про невдалу спробу поповнити рахунок та пропонує спробувати ще раз.

Оформлення підписки на видання

Основний успішний сценарій

  • 1. Авторизований користувач нажимає кнопку для здійснення підписки біля обраного видання з вказанням терміну в місяцях (обмеження – 12 місяців максимум).
  • 2. Система перевіряє, чи достатньо коштів на персональному рахунку користувача.
  • 3. Система підтвердила, що користувачеві вистачає коштів для підписки.
  • 4. Користувач підтверджує, що хоче здійснити підписку.
  • 5. Система списує повну плату за підписку з визначеним терміном з персонального рахунку користувача.
  • 6. Система додає підписку в особистий кабінет користувача.

Альтернативний сценарій

  • 1. Авторизований користувач нажимає кнопку для здійснення підписки біля обраного видання.
  • 2. Система перевіряє, чи достатньо коштів на персональному рахунку користувача.
  • 3. Система отримала інформацію, що користувачеві не вистачає коштів для підписки.
  • 4. Система сповіщає користувача про неможливість здійснення підписки через брак коштів та пропонує поповнити рахунок.

Перегляд своїх підписок

  • Користувач в особистому кабінеті бачить список своїх активних підписок та термін їх дії. Також користувач бачить загальну суму, яку він витратив на сервісі та кількість неактивних підписок.

Додати видання

Основний успішний сценарій

  • 1. Адміністратор вводить валідні значення полів для видання:
  • • назва – мінімум 2 символа, максимум 100 символів;
  • • опис – мінімум 10 символів, максимум 255 символів;
  • • вартість підписки за місяць – числове значення, більше 0.01;
  • • жанр – обраний зі списку.
  • Поля “назва” та “опис” мають по два обов’язкові для введення варіанти – на українській та на англійській мовах.
  • 2. Система додає видання до списку усіх видань.

Альтернативний сценарій

  • 1. Адміністратор ввів невалідні значення полів для видання.
  • 2. Система інформує користувача про невалідні вхідні дані.

Редагувати інформацію про видання

Основний успішний сценарій

  • 1. Адміністратор змінює поля, які вважає за потрібне згідно валідаційних норм для видання.
  • 2. Система зберігає зміни.

Альтернативний сценарій

  • 1. Адміністратор або нічого не змінює, або вводить невалідні значення для полів видання.
  • 2. Система відображає інформацію про помилку збереження змін.

Видалення видання

  • 1. Адміністратор нажимає кнопку для видалення видання.
  • 2. Система перепитує адміністратора, чи він впевнений.
  • 2.1. Якщо підтверджує скасування – система видаляє видання зі списку усіх видань та відбувається скасування усіх користувацьких підписок на видалене видання.

Перегляд статистичних даних

Основний сценарій

  • • Адміністратор має список користувачів та бачить, хто скільки активних та неактивних підписок має, а також суму, яку витратив користувач на підписки.
  • • Адміністратор має можливість завантажити дані в pdf-форматі.

Заблокувати користувача

Основний сценарій

  • 1. Адміністратор нажимає на кнопку блокування визначеного користувача.
  • 2. Система змінює стан облікового запису користувача на заблокований.

Розблокувати користувача

Основний сценарій

  • 1. Адміністратор нажимає на кнопку розблокування визначеного користувача.
  • 2. Система змінює стан облікового запису користувача на незаблокований.

Спеціальні вимоги

  • • Система повинна мати можливість перемикання мови інтерфейсу.
  • • Швидкий обмін даними між клієнтом і сервером, сервером та базою даних.

Імплементація шаблон Front Controller у вигляді класу FrontController

class FrontController extends HttpServlet {

    private static final long serialVersionUID = 2883474034122955977L;

    private static final Logger LOG = Logger.getLogger(FrontController.class);

    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
        LOG.debug("Front controller doGet starts");
        HttpPath httpPath = processRequest(req, resp);
        LOG.debug("Front controller doGet finished, now go to address with handling type --> "
                + httpPath.getPath() + " - " + httpPath.getHttpHandlerType());

        switch (httpPath.getHttpHandlerType()) {
            case FORWARD:
                RequestDispatcher disp = req.getRequestDispatcher(httpPath.getPath());
                disp.forward(req, resp);
                break;

            case SEND_REDIRECT:
                resp.sendRedirect(req.getContextPath() + httpPath.getPath());
                break;
        }
    }

    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws IOException {
        LOG.debug("Front controller doPost starts");
        HttpPath httpPath = processRequest(req, resp);
        LOG.debug("Front controller doPost finished, now sendRedirect to --> " + httpPath.getPath());

        resp.sendRedirect(req.getContextPath() + httpPath.getPath());
    }

    /*
        * Main method of this controller, that process all: Get and Post, - requests.
        *
    */
    private HttpPath processRequest(HttpServletRequest req, HttpServletResponse resp) {
        // extract command name from the request
        String commandName = req.getParameter("command");
        LOG.trace("Request parameter: command --> " + commandName);

        // obtain command object by its name
        Command command = CommandContext.getCommand(commandName);
        LOG.trace("Obtained command --> " + command);

        // execute command and get http handler type with path
        HttpPath httpPath = command.execute(req, resp);
        LOG.trace("HttpPath --> " + httpPath);
        return httpPath;
    }
}

Клас CommandContext для зберігання імплементованих команд (використовується шаблон Command)

public final class CommandContext {

    private static final Logger LOG = Logger.getLogger(CommandContext.class);

    private static final Map commandMap = new HashMap<>();

    static {
        commandMap.put("sign-in", new SignInPageCommand());
        commandMap.put("sign-up", new SignUpPageCommand());

        commandMap.put("registration", new RegistrationCommand());
        commandMap.put("login", new LoginCommand());
        commandMap.put("logout", new LogoutCommand());

        commandMap.put("catalog", new CatalogCommand());
        commandMap.put("payment", new PaymentCommand());
        commandMap.put("subscribe", new SubscribeCommand());
        commandMap.put("subscriptions", new SubscriptionsCommand());
      
        commandMap.put("users", new UsersManagementCommand());
        commandMap.put("generate_users_pdf", new GenerateUsersPdfCommand());
        commandMap.put("change_user_status", new ChangeUserStatusCommand());
        commandMap.put("publications", new PublicationsManagementCommand());
        commandMap.put("publication_delete", new PublicationDeleteCommand());
        commandMap.put("publication_edit_view", new PublicationEditViewCommand());
        commandMap.put("edit_publication", new EditPublicationCommand());
        commandMap.put("publication_create_view", new PublicationCreateViewCommand());
        commandMap.put("create_publication", new CreatePublicationCommand());

        LOG.debug("Command context was successfully initialized");
        LOG.trace("Number of commands --> " + commandMap.size());
    }

    private CommandContext() {}

    /*
        * Returns command object with the given name.
        *
        * @param commandName Name of the command.
        * @return Command object.
    */
    public static Command getCommand(String commandName) {
        if (commandName != null && commandMap.containsKey(commandName)) {
            return commandMap.get(commandName);
        }

        return null;
    }
}
